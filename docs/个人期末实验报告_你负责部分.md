## 摘要

本次期末实验围绕“基于淘宝用户消费行为的大数据分析与精准运营研究”展开。小组整体任务是利用淘宝用户消费行为数据、订单交易数据、用户画像数据和商品数据，从用户行为分析、用户价值分层、商品表现评估等角度完成一次完整的大数据分析实践。在小组分工中，我主要负责 MySQL 实现与 Spark / KMeans 用户聚类分析两部分内容，目标是补齐项目在结构化存储、多表关联查询以及机器学习分群方面的技术实现短板。

在 MySQL 部分，我围绕用户表、商品表、订单表和行为表设计了结构化数据库表结构，完成了建库建表 SQL、CSV 数据导入脚本以及多表关联查询示例。通过关系型建模，我将原始数据从文件形式映射为可查询、可关联、可复用的数据资源，为后续分析提供了结构化支撑。

在 Spark / KMeans 部分，我以用户消费行为特征为基础，构建了包含最近消费时间、订单频次、消费金额、浏览行为、点击行为、收藏行为、加购行为、订单质量特征和行为比例特征等在内的用户特征向量，并采用 Spark MLlib 中的 KMeans 算法进行用户聚类。实验进一步进行了不同 `k` 值对比和特征消融分析，最终发现移除 `engagement_score` 与 `member_level_score` 两个冗余特征后，聚类效果得到明显提升，并最终选择 `k=3` 作为最优聚类数，对应轮廓系数达到 `0.3542`。结合 RFM 分层结果对照分析发现，KMeans 聚类在高价值用户与流失风险用户识别上与 RFM 结果具有较高一致性：其中 Cluster 0 明显表现出高消费、高频次、高客单价的高价值特征，而 Cluster 1 则表现出消费周期长、订单少、消费低的流失风险特征。

本实验表明，MySQL 能够有效承担结构化存储与多表分析的基础角色，而 Spark / KMeans 能够作为规则分层的补充验证工具，从数据分布角度进一步识别用户价值层次。虽然当前聚类效果仍有继续优化空间，但整个实验已经实现了从原始数据到数据库建模，再到机器学习分群的较完整技术流程，为小组整体课题的完善提供了关键支撑。

## 一、绪论

### 1.1 研究背景

随着电子商务平台快速发展，用户在平台上的每一次浏览、点击、收藏、加购、下单和评价行为，都会沉淀为可分析的数据资产。对于平台来说，真正的问题并不是“有没有数据”，而是“能不能把数据转化为运营决策能力”。如果不能识别高价值用户，平台就无法合理分配营销资源；如果不能识别低活跃用户，平台就无法及时进行用户召回；如果不能判断商品是否具有高转化潜力，平台就无法优化选品和流量分配策略。

在传统分析方式中，很多企业仅依赖简单报表统计，如订单量、销售额和客单价等指标。这些指标虽然可以描述整体经营结果，却很难回答更深入的问题，例如：

- 哪类用户的长期价值最高？
- 哪些用户虽然现在消费不高，但具有上升潜力？
- 哪些用户已经表现出明显流失倾向？
- 用户行为与最终消费结果之间到底存在怎样的关联？

这些问题本质上要求我们从“结果统计”转向“行为理解”和“用户分群”。

在本课题中，小组整体以淘宝用户消费行为数据为基础，构建从数据存储、数据处理、离线分析到结果可视化的完整分析链路。而我负责的 MySQL 与 Spark / KMeans 两部分，分别对应项目中两个很重要的层次：

- MySQL 负责解决“数据如何结构化落地”的问题；
- Spark / KMeans 负责解决“如何从多维特征中自动发现用户群体”的问题。

因此，我负责的部分既是项目的数据基础设施补充，也是用户价值分析模型的重要技术支撑。

### 1.2 研究意义

从课程设计角度看，完成这一部分工作具有两方面意义。

第一，MySQL 实现补充了项目的结构化数据管理能力。仅依赖 CSV 文件虽然能够完成部分分析，但在多表关联、字段约束、查询复用和数据组织方面都存在明显局限。通过关系型数据库建模，可以更好地组织用户、商品、订单和行为数据之间的逻辑关系，也更符合企业实际应用场景。

第二，Spark / KMeans 聚类为用户价值分层提供了规则模型之外的补充验证。RFM 模型的优点在于可解释性强，但它本质上仍是规则分层。KMeans 则能够从多维特征中自动发现用户群体，从另一个角度验证用户价值结构是否真实存在于数据分布中。两者结合后，可以使用户分群结果更具说服力。

### 1.3 个人工作定位

在本项目分工中，我负责的是一条偏“技术实现”与“模型验证”的主线。

具体而言，我的任务不是单纯撰写结论，而是通过代码、脚本和模型真正把以下内容落地：

- 将原始 CSV 数据映射到 MySQL 数据库；
- 设计和实现多表结构与多表关联查询；
- 基于订单与行为数据构造用户特征；
- 使用 Spark 完成 KMeans 聚类与轮廓系数评估；
- 将 KMeans 聚类结果与 RFM 分层结果进行对照分析。

也就是说，我这一部分的工作目标可以概括为：

> 从数据工程和机器学习两个角度，增强整个课题在技术实现层面的完整性与可信度。

## 四、整体技术路线

### 4.1 MySQL 路线

MySQL 部分的技术路线可以概括为：

`CSV 数据 -> 表结构设计 -> MySQL 建库建表 -> 数据导入 -> 多表关联查询`

其中，重点不在“把文件导进去”，而在于建立清晰的数据关系：

- 用户与订单的关系；
- 商品与订单的关系；
- 用户行为与商品的关系；
- 用户画像与消费结果的关系。

### 4.2 Spark / KMeans 路线

Spark 部分的技术路线可以概括为：

`订单数据 + 行为数据 + 用户画像 -> 特征构造 -> Spark 向量化与标准化 -> KMeans 聚类 -> 轮廓系数评估 -> 与 RFM 对照分析`

该路线的核心思想是：

- 用更丰富的多维特征替代单纯依赖订单金额的粗粒度分层；
- 通过无监督聚类发现数据内部存在的用户结构；
- 再与规则分层进行比对，增强结果可信度。

## 五、MySQL 模块设计与实现

### 5.1 表结构设计原则

在 MySQL 部分，我优先考虑了以下几个设计原则：

1. 用户表和商品表作为维度表；
2. 订单表和行为表作为事实表；
3. 尽量保留原始数据字段语义；
4. 为后续关联查询预留主键、外键与索引结构。

基于这一思路，我最终构建了四张核心表：

- `users`
- `products`
- `orders`
- `user_behaviors`

图 5-1 展示了我为本项目设计的核心数据表实体关系图。从图中可以看出，`users` 与 `products` 作为维度表，`orders` 与 `user_behaviors` 作为事实表，通过 `user_id` 与 `product_id` 两类外键建立关联关系。这种结构既满足订单分析需求，也满足行为分析需求，为后续 MySQL 查询和 Spark 特征构造提供了统一的数据基础。

### 5.2 建库建表实现

我编写了 `docs/mysql/schema.sql` 用于完成建库建表。

设计上主要考虑了：

- `users.user_id`、`products.product_id` 作为主键；
- `orders` 中的 `user_id` 和 `product_id` 分别关联用户表和商品表；
- `user_behaviors` 中的 `user_id` 和 `product_id` 同样建立关联；
- 为 `order_date`、`behavior_time`、`behavior_type` 等字段建立索引。

这种设计有两个优势：

第一，方便后续使用 SQL 快速完成画像-订单-商品的交叉查询；

第二，更接近企业真实数据库建模方式，有利于将实验结果迁移到实际业务场景。

## 六、Spark / KMeans 模块设计与实现

### 6.1 特征工程设计

为了让 KMeans 聚类真正有意义，我没有简单只用 RFM 三个字段，而是扩展了更丰富的特征。

本次构造的特征包括：

- `recency_days`
- `order_count`
- `total_spent`
- `avg_order_amount`
- `total_quantity`
- `browse`
- `click`
- `favorite`
- `cart`
- `buy`
- `avg_duration_seconds`
- `member_level_score`

其中：

- `recency_days` 表示最近一次消费距当前参考日期的天数；
- `order_count` 表示订单频次；
- `total_spent` 表示累计消费金额；
- `avg_order_amount` 表示平均客单价；
- `browse/click/favorite/cart/buy` 表示行为强度；
- `avg_duration_seconds` 表示兴趣停留程度；
- `member_level_score` 表示会员等级数值化结果。

这样的特征体系比单纯使用 RFM 更完整，因为它不仅考虑消费结果，也考虑用户在购买前的行为过程。

### 6.2 RFM 标签构造

为了后续做对照分析，我在脚本中先用 Pandas 重建了 RFM 标签。

RFM 的规则为：

- R：最近消费时间越近越好；
- F：消费频次越多越好；
- M：消费金额越高越好。

在实现中，我通过五分位数对三个指标打分，并最终将用户划分为：

- 高价值用户
- 潜力用户
- 一般用户
- 流失风险用户

这一结果作为后续 KMeans 聚类的对照基准。

### 6.3 KMeans 聚类实现

在 Spark 部分，我使用：

- `VectorAssembler` 完成特征向量拼接；
- `StandardScaler` 完成特征标准化；
- `KMeans` 完成用户聚类；
- `ClusteringEvaluator` 计算轮廓系数。

本次实验首先对多个聚类数进行了比较，比较范围包括：

- `k = 3`
- `k = 4`
- `k = 5`
- `k = 6`

该设置主要考虑到：

- 与 RFM 中四类用户有一定可对照性；
- 聚类数过多不利于业务解释；
- 聚类数过少则可能导致高价值用户和中间用户混淆。

### 6.4 脚本运行与输出结果

本次脚本真实运行成功，输出如下结果文件：

- `final_kmeans_summary.json`
- `final_kmeans_cluster_summary.csv`
- `final_kmeans_rfm_vs_kmeans.csv`
- `final_kmeans_cluster_labels.csv`

其中：

- `summary` 文件记录聚类数、轮廓系数和特征列；
- `cluster_summary` 文件记录各聚类的平均特征；
- `rfm_vs_kmeans` 文件记录 RFM 标签与聚类标签的交叉结果；
- `cluster_labels` 文件记录每个用户的聚类类别。

在完成基础聚类脚本后，我进一步进行了不同 `k` 值的多轮试验，比较了 `k=3、4、5、6` 四组结果，并选择轮廓系数最高的聚类方案作为最终展示结果。

### 7.3 聚类特征均值分析

#### 7.3.1 Cluster 0：高价值核心群体

Cluster 0 在多个关键指标上都明显优于其他群体：

- 平均订单数：`4.32`
- 平均总消费金额：`23166.09`
- 平均客单价：`6627.94`
- 平均购买件数：`13.82`

这类用户具备以下典型特征：

- 消费频繁
- 累计消费金额高
- 单次订单金额也高
- 购买件数多

因此，Cluster 0 可以被视为本次聚类中的核心高价值用户群体。

#### 7.3.2 Cluster 1：流失风险群体

Cluster 1 的特征表现为：

- `recency_days` 较高：`65.68`
- 订单数较低：`2.44`
- 总消费金额较低：`4245.82`
- 单件价值较低：`589.05`

同时，在 RFM 对照中，Cluster 1 中流失风险用户数量最多，达到 `878`，说明该群体具备较明显的流失风险特征。

#### 7.3.3 Cluster 2：潜力成长群体

Cluster 2 的总体特征介于高价值群体与流失群体之间：

- 平均总消费金额：`6248.41`
- 平均订单数：`2.90`
- 收藏率和加购率相对更高
- 综合行为得分 `4.10`，高于另外两类

这说明 Cluster 2 属于较典型的潜力成长型用户群体，虽然当前消费水平不及高价值群体，但行为活跃度和转化信号较强，具备进一步运营激活的价值。

为了进一步比较三个聚类群体在多项核心指标上的差异，图 7-4 从平均订单数、平均消费金额和平均点击率三个维度展示了群体特征。可以看出，Cluster 0 在消费金额和订单数上占优，而 Cluster 2 在行为活跃度方面更具潜力。

![图7-4 KMeans聚类群体综合指标对比](figures/kmeans_cluster_metric_compare.png)

### 7.4 RFM 与 KMeans 对照分析

将 RFM 与 KMeans 结果交叉统计后，得到以下现象：

| Cluster | 一般用户 | 流失风险用户 | 潜力用户 | 高价值用户 |

| 1 | 657 | 132 | 707 | 191 |
| 2 | 215 | 77 | 215 | 440 |

#### 7.4.1 高价值用户识别一致性

Cluster 0 中有 `506` 名高价值用户，是所有聚类中高价值用户最多的一类。

这说明：

- KMeans 可以识别出与 RFM 高价值用户相近的核心群体；
- 高价值用户不仅能通过规则识别，也能在多维特征空间中形成相对独立的群体。

#### 7.4.2 流失风险用户识别一致性

Cluster 1 中有 `878` 名流失风险用户，是所有聚类中流失风险用户最多的一类。

这说明：

- KMeans 对低活跃、低消费、长周期未消费用户的识别能力较强；
- Cluster 3 在业务上具有非常清晰的召回价值。

#### 7.4.3 中间群体存在交叉

Cluster 2 中同时分布着：

- 一般用户 `385`
- 流失风险用户 `334`
- 潜力用户 `409`
- 高价值用户 `138`

这说明规则分层与聚类分群在中间型用户识别上存在交叉，聚类更适合刻画“潜在成长性”，而 RFM 更强调当前消费结果。

其原因在于：

- RFM 主要关注消费结果；
- KMeans 同时考虑行为活跃度、停留时长、会员等级等更丰富的特征。

因此，KMeans 对中间用户的描述更细，而 RFM 对极端群体的描述更直接。

图 7-5 以热力图形式展示了 RFM 与 KMeans 的对照结果。图中颜色越深，表示该聚类中对应标签的用户数量越多。可以明显看出：

- Cluster 0 与高价值用户高度对应；
- Cluster 1 与流失风险用户高度对应；
- Cluster 2 则承担了中间层与潜力层的混合特征。

![图7-5 RFM与KMeans对照热力图](figures/rfm_kmeans_heatmap.png)

### 7.5 对业务的启示

从我的实验结果出发，可以得到以下业务启示：

1. 高价值用户群体是真实存在的，而且具有较明显的消费强度特征；
2. 流失风险群体也能被稳定识别，适合做召回运营；
3. 中间用户群体内部差异较大，不能简单统一管理；
4. RFM 与 KMeans 最适合采用“规则分层 + 聚类验证”的组合方式，而不是二选一。
